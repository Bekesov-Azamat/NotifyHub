# 🚀 Notification Service

## 📖 Описание проекта

Тестовое задание на **Laravel 11 + PHP 8.2**.

Реализован сервис уведомлений с поддержкой нескольких каналов доставки, REST API, очередей, хранения истории уведомлений и дополнительной генерацией отчётов.

Проект построен с упором на **расширяемость**, **читаемость кода** и **разделение ответственности** между компонентами.

---

## 🎯 Цель проекта

Основная цель — продемонстрировать практические навыки разработки backend-сервисов на Laravel с использованием REST API, очередей, тестирования, статического анализа кода и принципов расширяемой архитектуры.

## ✅ Реализованный функционал

### 📬 Создание уведомления

Поддерживается создание уведомления с:

- ID пользователя
- каналом доставки
- текстом сообщения

При создании уведомление получает статус `pending` и отправляется в очередь.

---

### 📄 Получение статуса уведомления

Реализован API для получения текущего состояния уведомления.

**Поддерживаемые статусы:**

| Статус | Описание |
|---|---|
| `pending` | в ожидании обработки |
| `processing` | в процессе отправки |
| `sent` | успешно отправлено |
| `failed` | ошибка доставки |

---

### 📜 История уведомлений пользователя

Реализован просмотр истории уведомлений пользователя.

**Поддерживаются фильтры:**

- по статусу
- по каналу доставки

---

### 📡 Каналы отправки

Поддерживаются:

- ✉️ **Email**
- 💬 **Telegram**

> Реальная отправка намеренно заменена заглушками в соответствии с требованиями тестового задания.

Использованы отдельные реализации каналов через интерфейс, что позволяет **без изменения существующего кода** добавлять новые способы доставки.

**Например:**

- SMS
- Push
- WhatsApp
- WebSocket

---

### ⚙️ Очередь

После создания уведомления выполняется постановка **Job** в очередь.

Обработка происходит отдельным **Worker**.

Реализована повторная попытка доставки.

**При успешной обработке:**
- статус меняется на `sent`
- увеличивается количество попыток
- записывается время отправки

**При ошибке:**
- увеличивается количество попыток
- сохраняется текст ошибки
- после исчерпания попыток статус становится `failed`

---

## 📊 Дополнительное задание — генерация отчётов

Реализована генерация отчётов.

**Поддерживается:**

- создание задания на генерацию
- получение текущего статуса
- скачивание готового файла

Отчёт сохраняется во внутреннюю файловую систему. Поддерживается фильтрация по периоду.

**В отчёт входят:**

- пользователь
- период
- общее количество уведомлений
- общее количество ошибок
- количество уведомлений по каждому каналу
- количество ошибок по каждому каналу
- дата генерации

---

## 🏗️ Архитектурные решения

### 🔌 NotificationChannelInterface

Для каналов доставки использован отдельный интерфейс **`NotificationChannelInterface`**.

Каждый канал реализует собственную логику отправки.

Выбор реализации выполняется через **`NotificationChannelFactory`**.

Такое решение позволяет **расширять систему без изменения существующего кода**.

---

### 🧩 NotificationSenderService

Вся логика отправки вынесена в отдельный сервис.

**Job не содержит бизнес-логику** и занимается только вызовом сервиса.

Это упрощает:

- тестирование
- сопровождение
- расширение проекта

---

### ⏱️ Queue

Отправка уведомлений выполняется **асинхронно**.

Создание уведомления не блокируется временем обработки доставки.

---

### 📁 Report Job

Генерация отчётов также вынесена в отдельную очередь.

**Во время выполнения статус меняется:**

`pending` → `processing` → `completed` / `failed`

При ошибке причина сохраняется в `last_error`.

## 🚀 Как запустить проект

### 1. Клонировать репозиторий

```bash
git clone <repository>
cd notifyhub
```

### 2. Создать `.env`

```bash
cp .env.example .env
```

### 3. Запустить Docker

```bash
docker compose up -d --build
```

### 4. Установить зависимости

```bash
composer install
```

### 5. Сгенерировать ключ

```bash
php artisan key:generate
```

### 6. Выполнить миграции

```bash
php artisan migrate
```

### 7. Запустить Worker

```bash
php artisan queue:work
```

## 💡 Что было важно при реализации

Основной целью было не просто реализовать работающий REST API, а сделать код **расширяемым и поддерживаемым**.

Поэтому логика разделена между отдельными сервисами, каналами доставки, фабрикой и очередями, а все входные данные проходят через **FormRequest-валидацию**.

Особое внимание уделялось читаемости кода, статическому анализу, тестам и возможности дальнейшего масштабирования проекта.
